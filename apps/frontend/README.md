## Backend Repository - Thesis Project

## 📋 Project Description

This is the **local user dashboard** of the complete automated testing system. Its purpose is to provide a modern, intuitive web interface for managing Playwright + BDD testing projects generated by the generation engine.

## 🏗️ MVP Architecture

This dashboard is part of the **Local User Environment** in the full system architecture:

![MVP Architecture](arquitectura.png)

### Dashboard Components

- **Main Dashboard**
  - *Technologies*: React 18 + TypeScript + Vite
  - *Description*: Overview of project status, metrics, and recent runs

- **Project Management**
  - *Technologies*: React Router + TailwindCSS + Radix UI
  - *Description*: Interface to create, edit, and manage generated testing projects

- **Execution Center**
  - *Technologies*: WebSockets + Framer Motion
  - *Description*: Real-time monitoring of test runs with logs and metrics

- **Local Settings**
  - *Technologies*: Context API + LocalStorage
  - *Description*: Management of preferences, themes, and user-specific settings

## 🎯 Purpose of this MVP

This dashboard is the **local user interface** of the complete system. It focuses on providing a modern web experience for:

- **Managing projects** generated by the engine
- **Monitoring runs** in real time
- **Configuring** local user preferences
- **Viewing** testing metrics and reports

**Why a local web interface?**
- Provides a modern experience without native apps
- Runs in the user's browser with full privacy
- Communicates with the local generation engine via REST API
- Allows access from any device on the local network

## 🚀 Installation and Running

### Method 1: Direct Installation (⭐ RECOMMENDED)

#### 1. Clone the Repository
```bash
git clone https://github.com/BryanGaray99/omega-testing-dashboard-mvp.git
cd omega-testing-dashboard-mvp
```

#### 2. Configure Environment Variables
```bash
# Copy the example file
cp .env.example .env

# Edit variables according to your setup
# Especially important:
# - VITE_API_URL: Generation engine URL (default: http://localhost:3000)
# - VITE_APP_PORT: Dashboard port (default: 5173)
```

#### 3. Install Dependencies
```bash
npm install
```

#### 4. Run the Dashboard
```bash
npm run dev
```

#### 5. Verify It Works
- **Dashboard**: http://localhost:5173
- **Generation Engine**: http://localhost:3000 (must be running)

### ⚠️ .env File Configuration

Make sure your `.env` file contains the following variables:

```env
# Generation engine URL (backend)
VITE_API_URL=http://localhost:3000

# Dashboard port
VITE_APP_PORT=5173

# Development mode
VITE_DEV_MODE=true

# Authentication configuration (future)
VITE_AUTH_ENABLED=false

# Theme configuration
VITE_DEFAULT_THEME=system
```

### Method 2: With Docker (Development/testing only)

```bash
docker build -t omega-dashboard .
docker run -p 5173:5173 omega-dashboard
```

## 🧪 Testing the Dashboard

All tests are run from the terminal from `apps/frontend`.

### 1. Unit / component accessibility (Vitest + vitest-axe)

Covers **all main screens** (Layout, Dashboard, Projects, Endpoints, Test Cases, Test Suites, Bugs, Test Executions, Reports, Logs, AI Assistant, Settings: Documentation, Appearance, OpenAI, Danger Zone). Each test renders the page and asserts no axe violations in the DOM (labels, ARIA, structure). Contrast is not evaluated in jsdom.

**Run from terminal:**

```bash
cd apps/frontend
npm test
```

### 2. E2E accessibility (Playwright + @axe-core/playwright)

Same set of routes in a real browser: axe scan per route plus 1–2 short flows (e.g. open Projects, open Settings → Appearance). Some axe rules are disabled for a stable baseline (e.g. color-contrast, region, button-name).

**Run from terminal:**

```bash
cd apps/frontend
npx playwright install   # first time only
npm run test:e2e
```

### 3. Lighthouse audits (Performance, Accessibility, Best Practices, SEO)

Runs **Google Lighthouse** on all main screens (Dashboard, Projects, Endpoints, Test Cases, Test Suites, Bugs, Test Executions, Reports, Logs, AI Assistant, Settings subpages). Same four categories as in Chrome DevTools:

- **Performance** – FCP, LCP, TBT, CLS, Speed Index (threshold: 50)
- **Accessibility** – automated a11y checks (threshold: 90)
- **Best Practices** – CSP, source maps, etc. (threshold: 85)
- **SEO** – meta description, title, crawlability, etc. (threshold: 50)

HTML reports are written to `apps/frontend/lighthouse-reports/`. Runs with a single worker; the dev server is started automatically if not already running.

**Run from terminal:**

```bash
cd apps/frontend
npm run test:lighthouse
```

### 4. Usability

Broader usability (clarity, satisfaction, learnability) is not fully covered by the above and still benefits from manual testing.

## 🛠️ Troubleshooting

### Port in use
The dashboard automatically tries ports 5173, 5174, and 5175. If all are in use:
```bash
VITE_APP_PORT=5176 npm run dev
```

### Dependency issues
```bash
npm cache clean --force
rm -rf node_modules package-lock.json
npm install
```

### Generation engine not available
Make sure the generation engine is running:
```bash
# In another terminal, from the backend directory
cd ../central-backend-mvp
npm run start:dev
```

### API configuration
If the generation engine is on a different port:
```bash
# In .env
VITE_API_URL=http://localhost:3001
```
